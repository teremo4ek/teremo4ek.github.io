After years of writing Go services in production, I've settled on a few error handling patterns that keep code readable and debugging painless.

## Wrap, Don't Lose Context

The single most important rule: always wrap errors with context. A bare `return err` tells you nothing when it surfaces in logs three layers up.

```go
result, err := db.Query(ctx, query)
if err != nil {
    return fmt.Errorf("fetching user by id %d: %w", id, err)
}
```

The `%w` verb enables `errors.Is` and `errors.As` checks at the call site — use it consistently.

## Sentinel Errors for Public APIs

Define expected error values for your package boundaries:

```go
var ErrNotFound = errors.New("resource not found")
```

Callers can then branch on behavior, not error messages:

```go
if errors.Is(err, pkg.ErrNotFound) {
    // handle missing resource
}
```

## Custom Error Types for Rich Context

When you need structured data attached to an error, a custom type beats string formatting:

```go
type ValidationError struct {
    Field   string
    Message string
}

func (e *ValidationError) Error() string {
    return fmt.Sprintf("validation failed: %s — %s", e.Field, e.Message)
}
```

## Keep It Simple

Not every error needs a custom type or a sentinel. Most of the time, `fmt.Errorf` with `%w` is enough. Reach for more structure only when callers need to act on the error programmatically.
